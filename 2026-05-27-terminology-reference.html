1. Architectural layers — the hierarchy

Orchestrator
   │
   ├── 3 Gates    (margin, platform_risk, quality_compliance_triage)
   ├── 1 Critic   (final reviewer, post-plan)
   │
   └── 8 Coordinators
          └── ~22 Agents   (Tier A: 14, Tier B: 8; Tier C deferred)
Term Definition Cardinality Repo equivalent
Orchestrator The single top-level dispatcher. Builds the holistic plan before any action; carries data between coordinators. Krell's analogy: "general manager" — broad, not deep. 1 Orchestrator (same) — src/agents/orchestrator.py
Coordinator Middle-layer dispatcher under the Orchestrator. Prompt only — no runbooks, no tools. Decomposes an attributed job into agent tasks, dispatches to its specialist agents, assembles results. Coordinators do not call each other. 8 New in v2. No current equivalent; pending glossary issue.
Agent A specialist that does one domain's work. Equipped with prompt + one or more runbooks + tools. Reports back to its coordinator only. ~22
14 A 8 B		 Sub-Agent — SubAgentBase subclass under src/agents/implementations/<domain>.py
Gate A specialist agent the Orchestrator consults as minimum context before building the plan. Currently: margin, platform_risk, quality_compliance_triage. 3 No distinct repo term — gates are Sub-Agents with a specific pre-plan role.
Critic A final reviewer. Quality-checks the Orchestrator's assembled answer after all sub-agents have run. Not a gate. Rejection triggers one Report retry; sub-agents are not re-run. 1 Critic — src/agents/critic.py (same)

2. On-disk artefacts — what writers produce

Term Definition Where it lives in repo Notes
Runbook A procedural file an agent loads. Step-by-step "what to do" in a specific situation. Must contain the agent ID + coordinator ID the runbook belongs to. docs/runbooks/<name>.md v2 design: an agent has one or more runbooks. (Repo glossary today says "a runbook" singular — being updated.)
Prompt
(system prompt)		 A file that defines who the agent (or coordinator) is — role, posture, scope. Loaded into every LLM call. src/agents/prompts/<name>.md "Prompt = who you are. Runbook = what to do." Don't confuse them.
Tool An external action the agent can request — typically an API call or aggregation query. Each tool = one Hub HTTP endpoint. Hub endpoints under src/functions_hub/; tool wiring in src/agents/ Tools are the agent's verbs. The runbook tells the agent when to use which tool.
Red-line evaluation The acceptance gate for an agent or coordinator. Format: checklist or pass/fail rubric. Required for every runbook and coordinator prompt. Template: Krell's Campaign Architecture runbook. Inline in each runbook / coordinator prompt In repo terms: combines Bounds (src/agents/safety/bounds.py), Kill-switch (src/agents/safety/killswitch.py), and Audit (docs/AUDIT_ARCHITECTURE.md).

3. Process & priority

Term Definition
The Flow The communication contract between layers — how agents talk to the coordinator, coordinator to orchestrator, orchestrator to client, and back. Concept, not a runbook. Originated by Steven + Sun; domain experts supply query examples. Flow comes before runbook detail.
Tier A / B / C Writing priority, not architectural rank. Tier A (14 agents) + Tier B (8 agents) ship by 2026-06-03. Tier C (8 agents) is deferred until a necessity is identified.
Plan call The Orchestrator's first LLM call. Produces a JSON plan of steps {agent, goal, notes}.
Report call The Orchestrator's second LLM call. After sub-agents run, summarises results for the user. Reviewed by the Critic.

4. Communication rules (locked)

Non-negotiable for this design round. These govern who talks to whom.

  1. Agents talk to their own coordinator only. Never to another agent. Never to another coordinator. Never directly to the Orchestrator.
  2. Coordinators talk to the Orchestrator only. Never to another coordinator.
  3. The Orchestrator builds the full plan upfront, before any action. Sub-agents do not act before the plan exists.
  4. Missing information path: Agent → its coordinator → Orchestrator (if the coordinator can't resolve it).
